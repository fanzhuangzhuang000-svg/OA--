### P0 stock_records 迁移幂等性加固 ✅

**动机**: v0.3.14 全量部署时 stock_records 4 个 migration 跑过 (列已存在)。
v0.3.15+ 任何环境重跑会报 `1060 Duplicate column` 或 PG `42P07`。
**对策**: 4 个 migration 加 `Schema::hasColumn/hasTable` 守卫。

**改的 3 个文件** (down() 也加守卫):
1. `2024_01_07_000003_create_stock_records_table.php` — `hasTable` 守卫
2. `2026_06_21_130000_add_party_fields_to_stock_records.php` — `hasColumn('party_type')` 守卫
3. `2026_06_21_140000_add_logistics_to_stock_records.php` — `hasColumn('logistics_company')` 守卫

**第 4 个保留 no-op**:
- `2026_06_16_120000_extend_stock_records_type_enum.php` — PG 实际是 varchar(20)，ENUM 语法在 PG 不可用，留 no-op + 注释

### 烟囱测试 (`.workbuddy/_test/test_stock_idempotent_v0317.py`)

4 步流程:
1. 备份当前 schema (列+索引)
2. **DELETE migrations 记录** (不走 rollback，防外键 cascade)
3. 第 1 次 `migrate` → 必须 DONE
4. 第 2 次 `migrate` → 必须 `INFO Nothing to migrate.`

**结果**:
- 172 第 1 次: 5 migration 全部 DONE (5.36ms / 1.77ms / 2.51ms)
- 172 第 2 次: `INFO Nothing to migrate.` ✅ (守卫生效)
- 列 20 个 / 索引 10 个 两次完全一致
