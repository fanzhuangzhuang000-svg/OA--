## 踩坑
### 永久角色被覆盖成临时的灾难逻辑
V0.5.3 第一次写 helper 时, grant() 走"已存在就 UPDATE expires_at"路径 — 当 user 已永久持有某角色, 再授临时版本会把永久那条 UPDATE 成 expires_at IS NOT NULL, 永久角色变临时, 业务灾难。
**修**: grant() 检查 existing.expires_at IS NULL 时抛 LogicException, 要求先 revoke 再授临时。
**测试**: tearDown 必须重插 sales_yang 的 manager 永久, 否则后续测试用 sales_yang 测新角色会全失败。

### 部署 117 /var/www/oa-api 是 www-data 拥有
nbcy 没写权。deploy_v053.py 第一版用 sftp.put 直接写失败, 改走 /tmp + sudo cp + chown + chmod 三步。

### 本地 127.0.0.1:8081 访问限制
Windows 端 smoke 用 127.0.0.1:8081 需 SSH 端口转发, 转发进程会被回收导致连不上。改用 192.168.3.117:80 (nginx proxy 到 8081) 即可。

### phpunit 跑在 117 上但 doFlush 走 ssh redis-cli
失败 (phpunit 用 www-data 跑, 没法 ssh 到 nbcy)。
**修**: 改用 php-redis 直连本地 127.0.0.1:6379, select(0) select(1) FLUSHDB。

### .phpunit.cache 权限
root 用户跑过 phpunit 后 www-data 写不进去。修: `sudo chown -R www-data:www-data /var/www/oa-api/.phpunit.cache`, 之后 `sudo -u www-data php vendor/bin/phpunit`。

### User::activeRoles() / hasActivePermissionTo() vs spatie cache
spatie 5min cache 会让 hasRole/hasPermissionTo 返回过期数据。所有 V0.5.3 active 方法走 raw query + role_has_permissions join, 绕开 cache。

### V0.5.3 controller 没挂 permission:system.role 中间件
第一次部署完 eng_qian (manager 角色) 也能授临时角色, 应 403。修: 5 个新路由都加 `->middleware('permission:system.role')`。

### 测试 ID 错位
写测试时用了 hardcoded `/users/2/...` 但 fin_wu id=80, eng_qian id=82, sales_yang id=86。统一改成 admin token + dynamic id 查询 (通过 /users?keyword=eng_qian)。

### 路由顺序
`POST /api/users/{user}/roles/temporary` 必须在 `GET /api/users/{user}` 通配之前, 否则会被 user controller 捕获。

## 关键决策
- **临时角色 vs 永久角色** 走不同 product: "临时 = expires_at IS NOT NULL", spatie 单一 pkey 限制下, 同一用户同角色只能 1 行 (要么永久要么临时, 不能并存)。业务规则: 已有永久不能再授临时, 需先 revoke。
- **cleanExpired** 走每日 cron (00:30), 过期行物理删除 + 7 天内即将过期写站内通知。
- **expiringSoon** 默认 7 天, dashboard 用。
