## v0.3.7.5 路由 404 修复

### 根因
- 11 个父路由（attendance/employee/customer/project/service/expense/vehicle/inventory/finance/knowledge/message/settings）设了 `redirect` 到子路径
- 但代码里所有 `router.push('/attendance')` / `router.push('/project')` 都是简写
- 简写没有 alias → 命中 `/:pathMatch(.*)*` → 404

### 修复
- 给 11 个父路由加 `alias: '/xxx'` 指向自己
- 简写和子路径都能命中

### 验证工具

- `.workbuddy/e2e_routes.py`：实际 curl 测试 60 个业务路由，全 200

### 教训
- 写 redirect 父路由时**必须同时加 alias**，否则子路径跳转全 404
- 用 alias 比改全部代码 router.push 路径省事
- 静态扫描脚本（audit_routes.py）能发现肉眼看不出的 404
- 别用 grep 的 `\{.*?\}` 配嵌套括号，要手写括号计数

## v0.3.9.1 数据大屏 500 修复 (2026-06-17 14:00)

### 症状
- 浏览器访问数据大屏：`Server Error` + `大屏数据加载失败: Request failed with status code 500`
- POST /api/dashboard/screen → 500
- log: `SQLSTATE[42703]: Undefined column: 7 ERROR: column "hour" does not exist`

### 根因
- `DashboardController::serviceMetrics()` 用了 MySQL 专属的 `TIMESTAMPDIFF(HOUR, x, y)` 和 `TIMESTAMPDIFF(MINUTE, x, y)`
- PG 没有这个函数

### 修复（两处）
1. `TIMESTAMPDIFF(HOUR, assigned_at, completed_at) <= sla_hours`
   → `EXTRACT(EPOCH FROM (completed_at - assigned_at)) / 3600 <= sla_hours`
2. `AVG(TIMESTAMPDIFF(MINUTE, assigned_at, started_at))`
   → `AVG(EXTRACT(EPOCH FROM (started_at - assigned_at)) / 60)`

### 教训
- PG 算时间差用 `EXTRACT(EPOCH FROM (later - earlier)) / N`（秒数→分钟/小时/天）
- MySQL 特有函数：TIMESTAMPDIFF / DATE_FORMAT / DATEDIFF / IFNULL → 都要换成 PG 写法
- 应该一次性全局扫：`grep -rn "TIMESTAMPDIFF\|DATE_FORMAT" app/` 找出所有 MySQL-isms
- 部署脚本: `.workbuddy/newserver/fix_dashboard_pg.py`
- 系统 Chrome 路径 `C:/Program Files\Google\Chrome\Application\chrome.exe` 可用，不需要下载 playwright 自带浏览器

## v0.3.12 增量修复 (2026-06-17 16:12)

### 用户报"项目详情服务器内部错误"——根因不是后端 500

**真相**：用户看到的是 `axios` 拦截器对 5xx 报的通用 toast 消息"服务器内部错误"，但实际是**浏览器加载 JS chunk 时 nginx 返回 404**。

**问题链**：
1. 我加 `vite.config.ts` 的 `manualChunks` 重新拆 chunk
2. chunk 名字从单 `index-XXX.js` 变成 `Leave-XXX.js` + `MainLayout-XXX.js` 等
3. 用户浏览器还缓存着旧 HTML（nginx 默认 `location ~* \.(js|css|...)$` 设了 `expires 7d` + 静态资源 7d 缓存）
4. 旧 HTML 引用旧 chunk 名 → 浏览器请求 `Leave-DN_a4k3d.js` → nginx 404
5. 用户看到 "服务器内部错误" 误以为是 API 500
6. 我反复打 API 测都 200 → 找不到问题
7. **真正线索是 nginx error.log**: `open() "/var/www/oa-web/assets/Leave-DN_a4k3d.js" failed (2: No such file or directory)`

1. **回滚 vite manualChunks** → 单 chunk `index-CHiIXyzC.js`（牺牲一些首屏速度，换稳定）
2. **nginx 给 JS/CSS 加 SPA fallback**:
   ```nginx
   location ~* \.(js|css|...)$ {
       try_files $uri $uri/ /index.html;  # 找不到就回 index.html
       expires 7d;
   }
   ```
3. **index.html 不缓存**:
   ```nginx
   location = /index.html {
       add_header Cache-Control "no-cache, no-store, must-revalidate" always;
   }
   ```

### 关键教训
- **nginx error.log 永远是第一线索**（先看 404 而不是只看 application log）
- **JS chunk hash 变化 + 浏览器缓存 = 404 雪崩**
- **SPA fallback 必须覆盖 JS/CSS**，不只是 path-only 的 SPA 路由
- **`index.html` 必须 no-cache**，否则 hash 变了浏览器不知道
- **axios 拦截器对 5xx 报的"服务器内部错误"是误导**，要看 network 真实 status

### 调试流程（防止重犯）
1. 用户报"服务器内部错误" → **先看 network**，确认是 500 还是 404
2. nginx error.log 比 application log 更早暴露问题
3. 改 chunk 拆法时**必须**也改 nginx 缓存策略，否则用户会一直 404

## v0.3.12.1 NaN/undefined 500 防御 (2026-06-17 16:22)

### 用户报"项目详情 服务器内部错误"——Network 截图
F12 截图显示 `NaN`/`construction-logs?per_page=...`/`contracts`/`suppliers` 全部 500。

### 真正的 4 个根因

1. **laravel.log 错误**：
   ```
   production.ERROR: Unsupported operand types: string * int
   at vendor/laravel/framework/src/Illuminate/Database/Query/Builder.php:2780
   ```
   - 原因：`paginate('NaN')` → forPage() 内部 `($page - 1) * $perPage`，`$perPage = 'NaN'` 字符串，字符串 * int 报错

2. **NaN 从哪里来**：
   - 前端 `useRoute().query` 拿到的 `page`/`per_page` 可能是字符串 'undefined'/'null'/'NaN'
   - axios 默认会把 JS `undefined`/`null` 序列化成字符串
   - 比如 `get(url, { per_page: someVar })`，someVar 是 undefined 时 → `?per_page=undefined`

### 修复（双端防御）

**后端 ProjectController.php 3 处**：
```php
// 之前: paginate($request->per_page ?? 15)
// 之后: 强转 int + 边界保护
$perPage = (int) ($request->per_page ?? 15);
$perPage = max(1, min($perPage, 200));  // 限制 1-200
$query->paginate($perPage);
```

**前端 request.ts 加拦截器**：
```ts
// 清除 params 里无效值（undefined/NaN/null/'undefined'/'null'/'NaN'）
if (config.params && typeof config.params === 'object') {
  const cleaned: Record<string, any> = {}
  for (const [k, v] of Object.entries(config.params)) {
    if (v === undefined || v === null) continue
    if (typeof v === 'number' && Number.isNaN(v)) continue
    if (typeof v === 'string' && (v === '' || v === 'undefined' || v === 'null' || v === 'NaN')) continue
    cleaned[k] = v
  }
  config.params = cleaned
}
```

### 验证（11 个边界用例全 200）
- 正常 / NaN / undefined / null / 负数 / 超大 / 空
- NaN page / 空 page / contracts / suppliers

### 关键教训
- **`axios` 默认会序列化 `undefined` → 字符串 'undefined'**，不是忽略！
- **后端 paginate() 必须 (int) 强转 + 边界保护**，不能信任何 user input
- **NaN 错误排查**：
  1. 第一时间看 laravel.log 找堆栈
  2. 用浏览器 devtools network 抓真实请求 URL（包括 query string）
  3. 复现用 cURL 模拟带同样 query string

## v0.3.12.2 PG 22P02 防御 (2026-06-17 16:30)

**F12 截图**显示 `NaN/construction-logs/contracts/suppliers` 全部 500。

```

6. 之前 (int) 强转只防御了 `per_page` 等 query string，**没防御 URL path 里的 id**
7. 路由 `{project}` 模式 (implicit binding) 直接把 "NaN" 传给 PG

### 修复 — 三层防御

**第一层 (之前的修复) — 拦截器**：

$query->paginate($perPage);
```

**第三层 (新加) — 全局 PG 22P02 防御**：
`bootstrap/app.php` 加 QueryException 拦截：
```php
if ($e instanceof \Illuminate\Database\QueryException) {
    $sqlState = $e->errorInfo[0] ?? '';
    if ($sqlState === '22P02') {
        return response()->json([
            'code'    => 404,
            'message' => '资源不存在或参数错误',
        ], 404);
    }
}
```

- 正常: 200
- NaN/abc/字符串空格: 404 "资源不存在或参数错误"
- 0/-1/超大: 404 "接口不存在"
- 整型范围外: 404

### 教训
- **PG 22P02 是 implicit binding 的常见坑** — Laravel 不强转，PG 会拒绝
- **三层防御**：
  1. 前端拦截器清垃圾值
  2. Controller 强转 + 边界保护
  3. 全局异常处理兜底 5xx → 4xx

- **Laravel implicit binding 对 `{project}` 不会强转**：
  - `Project::find('NaN')` → 走 SQL → 22P02
  - `Project::find('abc')` → 走 SQL → 22P02
  - `Project::find('1')` → OK（PG 隐式转）
